# 草果剪贴板 - 测试指南

## 准备工作

### 1. 构建应用

```bash
# 构建 release 版本
npm run tauri build

# 或者运行开发模式（推荐）
npm run tauri dev
```

### 2. 启动两个应用实例

由于 Tauri 应用默认只能运行一个实例，我们需要使用不同的配置来启动两个实例。

#### 方法一：使用不同的配置文件（推荐）

1. 复制 `tauri.conf.json` 创建两个版本：

```bash
# 实例 1 配置
cp src-tauri/tauri.conf.json src-tauri/tauri-instance1.json

# 实例 2 配置
cp src-tauri/tauri.conf.json src-tauri/tauri-instance2.json
```

2. 修改实例 2 的配置，更改 `identifier` 和 `productName`：

```json
// tauri-instance2.json
{
  "$schema": "https://schema.tauri.app/config/2",
  "productName": "草果剪贴板-实例2",
  "version": "0.1.0",
  "identifier": "com.yangtanfang.caoguo-clipboard.instance2",
  ...
}
```

#### 方法二：在不同设备上运行（最佳）

- 在同一局域网的两台设备上分别运行应用
- 或者在一台设备上使用虚拟机或 Docker 容器

#### 方法三：使用开发模式 + 终端手动启动

```bash
# 终端 1: 启动实例 1
cd /Users/yangtanfang/project/2026/4cool/clipboard-caoguo
npm run tauri dev

# 终端 2: 启动实例 2（需要修改端口配置）
# 需要先修改 src-tauri/src/server/mod.rs 中的端口配置
```

## 测试步骤

### 测试 1: 应用启动和基本功能

**步骤：**
1. 启动第一个应用实例（实例 A）
2. 观察控制台输出，确认：
   - 设备 ID 生成成功
   - HTTP 服务器启动成功（端口 54321）
   - 数据库初始化成功
   - 剪贴板监听启动成功

**预期结果：**
```
📱 Device initialized:
   ID: <UUID>
   Name: <hostname>
✓ HTTP server started on port 54321
```

**验证点：**
- [ ] 应用窗口正常显示
- [ ] 剪贴板历史列表为空或显示已有记录
- [ ] 按 Cmd+Shift+V (macOS) 可以隐藏/显示窗口
- [ ] 窗口始终置顶

---

### 测试 2: mDNS 设备发现

**步骤：**
1. 确保实例 A 已经启动
2. 点击顶部的 "设备" 按钮
3. 确认 "启用设备发现" 开关已自动开启
4. 观察设备列表是否显示本机设备

**预期结果：**
- 本机设备应显示在设备列表中（状态为"已配对"）
- 控制台输出 mDNS 浏览启动信息

**验证点：**
- [ ] 设备列表显示本机设备
- [ ] 设备名称、IP 地址、端口显示正确
- [ ] "已配对" 状态显示正确

---

### 测试 3: 双实例发现

**步骤：**
1. 启动第二个应用实例（实例 B）- 使用不同的设备 ID
2. 等待 5-10 秒让 mDNS 发现完成
3. 在两个实例中分别打开设备面板
4. 检查设备列表

**预期结果：**
- 实例 A 的设备列表中应显示实例 B（状态为"未配对"）
- 实例 B 的设备列表中应显示实例 A（状态为"未配对"）

**验证点：**
- [ ] 实例 A 发现实例 B
- [ ] 实例 B 发现实例 A
- [ ] 设备信息（IP、端口）显示正确
- [ ] 状态显示为"未配对"

---

### 测试 4: 配对流程

**步骤：**
1. 在实例 A 中，找到实例 B 的设备
2. 点击"配对"按钮
3. 在实例 B 的设备面板中，查看是否有配对请求弹出
4. 在实例 B 中点击"接受"按钮
5. 查看两个实例的设备列表状态

**预期结果：**
- 实例 B 收到配对请求提示（通知或对话框）
- 接受后，两个实例的设备状态都变为"已配对"
- 两个实例的信任设备列表中都包含对方

**验证点：**
- [ ] 实例 B 收到配对请求
- [ ] 点击接受后状态变为"已配对"
- [ ] 两个实例都可以看到对方为已配对设备
- [ ] 信任设备列表正确更新

---

### 测试 5: 文本同步测试

**步骤：**
1. 确保两个实例已配对
2. 在实例 A 的系统剪贴板中复制一段文本
   ```bash
   echo "测试文本同步 - $(date)" | pbcopy  # macOS
   ```
3. 在实例 A 的应用中查看剪贴板历史
4. 等待 3-5 秒
5. 在实例 B 的应用中查看剪贴板历史

**预期结果：**
- 实例 A 的剪贴板历史显示刚复制的文本
- 实例 B 的剪贴板历史也显示同样的文本
- 时间戳正确

**验证点：**
- [ ] 实例 A 显示本地剪贴板内容
- [ ] 实例 B 自动同步到实例 A 的剪贴板内容
- [ ] 内容、时间戳完全一致

---

### 测试 6: 双向文本同步

**步骤：**
1. 在实例 B 复制文本：
   ```bash
   echo "从实例B同步 - $(date)" | pbcopy
   ```
2. 等待 3-5 秒
3. 在实例 A 中查看剪贴板历史
4. 在实例 A 复制另一段文本：
   ```bash
   echo "从实例A同步 - $(date)" | pbcopy
   ```
5. 等待 3-5 秒
6. 在实例 B 中查看剪贴板历史

**预期结果：**
- 两个实例都能同步对方的剪贴板内容
- 剪贴板历史列表中包含所有复制的内容

**验证点：**
- [ ] 实例 A 收到实例 B 的剪贴板内容
- [ ] 实例 B 收到实例 A 的剪贴板内容
- [ ] 双向同步正常工作
- [ ] 没有重复内容

---

### 测试 7: 文件传输测试

2. 点击"选择文件"按钮

**预期结果：**
- 实例 A 显示传输进度
- 传输完成后显示成功状态
- 实例 B 的传输历史显示接收记录
- 文件正确保存到下载目录

**验证点：**
- [ ] 文件选择正常
- [ ] 传输进度显示
- [ ] 传输成功完成
- [ ] 实例 B 收到文件
- [ ] 文件内容完整（MD5 校验）
- [ ] 可以打开文件目录查看

- [ ] 设备离线检测正常
- [ ] 设备重新上线被检测到

## 常见问题排查

### 问题：无法发现其他设备

**可能原因：**
1. 防火墙阻止 mDNS 流量
2. 设备不在同一局域网
3. mDNS 服务未启动

**解决方案：**
```bash
# macOS: 检查防火墙设置
# 系统偏好设置 -> 安全性与隐私 -> 防火墙 -> 防火墙选项
# 确保 mDNS 流量允许

# Linux: 安装 avahi
sudo apt-get install avahi-daemon

# 检查网络连接
ping <对方设备IP>
```

### 问题：配对请求未收到

**可能原因：**
1. HTTP 端口被阻止
2. 设备 ID 不匹配

**解决方案：**
```bash
# 测试 HTTP 连接
curl http://<对方设备IP>:54321/ping

# 检查设备 ID
# 在应用中查看 "关于" 页面或控制台日志
```

### 问题：剪贴板不同步

**可能原因：**
1. 设备未配对
2. 设备离线
3. 同步服务未启动

**解决方案：**
- 检查设备配对状态
- ping 对方设备确认在线
- 查看应用日志确认同步服务运行

### 问题：文件传输失败

**可能原因：**
1. 文件太大
2. 权限问题
3. 磁盘空间不足

**解决方案：**
- 检查磁盘空间
- 检查下载目录权限
- 尝试传输小文件
